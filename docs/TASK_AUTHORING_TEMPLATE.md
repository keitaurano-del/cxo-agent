# TASK_AUTHORING_TEMPLATE — Apollo タスク起票テンプレ／必須欄ルール

Apollo タスクボード（各プロジェクトの `docs/TASK_TRACKER.md`）にタスクを起票するときの標準テンプレートと必須欄ルール。task-manager が起票・構造化の正本として使う。

- 由来: MC-82（Keita 投入原文「アポロのタスクボードのタスクは何をやっているか詳細を記載すること → 成果物完了までのワークフローも記載すること」）の DoD(1)(2)。
- 表示面（Apollo TaskDetail での詳細メモ表示）は MC-83 で実装済み。本テンプレはその「書く側」を定義する。
- 対象台帳: `cxo-agent/docs/TASK_TRACKER.md`（MC-xx）/ `logic/docs/TASK_TRACKER.md` / `en-chakai/docs/TASK_TRACKER.md`（EC-xx）/ 西丸町 ほか TASK_SOURCES 全部。

---

## 1. 目的

Apollo の TaskDetail を開けば、誰が見ても「(a) このタスクは今何をやっているのか（詳細）」と「(b) 成果物が完了するまでの段取り（完了までのワークフロー）」が分かる状態を常に保つこと。

- (a) 詳細: そのタスクが何を解決しようとしているか、背景・現状・着地イメージ。
- (b) 完了までのワークフロー: 誰が → 何を → どの品質ゲートを通って → DONE になるかの段取り。

これらが欠けていると、TaskDetail に薄い一文しか出ず、進捗も段取りも読めない（MC-83 で見やすく表示しても中身が無ければ意味がない）。本テンプレは「中身を必ず書く」ための必須欄ルール。

**設計方針（重要）**: 新フィールドを乱立させない。既存台帳フォーマットの欄（詳細・受け入れ条件・サブタスク・次アクション・担当・依存）を徹底活用してテンプレ化する。「完了までのワークフロー」は新欄を足さず、既存の サブタスク（チェックリスト）＋ 次アクション ＋ 担当 ＋ 品質ゲートの明示で表現する。既存カードとの互換を壊さない。

---

## 2. 必須欄一覧

縦型カード（`| フィールド | 値 |`）の各欄の意味と、(a)詳細・(b)完了までのワークフロー がどの欄で表現されるかの対応表。

| 欄 | 必須/任意 | 意味 | (a)詳細 / (b)ワークフロー の対応 |
|---|---|---|---|
| ID | 必須 | タスク識別子。採番は `next-task-id.sh`（後述） | — |
| タイトル | 必須 | 一行で何のタスクか。動詞句で着地が分かる形に | — |
| 優先度 | 必須 | P0（最優先）〜P3（最低）。autonomous-rin は高い順に拾う | — |
| ステータス | 必須 | TODO / IN_PROGRESS / BLOCKED / REVIEW / DONE / CANCELLED | — |
| 担当 | 必須 | 実行エージェント（dev-logic/designer/content-creator 等）。複数なら役割を併記 | **(b) 誰が** の起点 |
| 詳細 | 必須 | 背景・現状・何を解決するか・着地イメージ。Keita 投入原文があれば引用 | **(a) 詳細の本体** |
| 関連 | 任意 | 関連ファイル（file:line）・関連タスク・参照 doc | (a) の裏取り根拠 |
| 受け入れ条件（DoD） | 必須 | 完了の定義。番号付き（1）（2）… で検証可能な形に | **(b) DONE の判定基準** |
| 依存 | 任意 | 前提となるタスク・着手条件。無ければ「なし」 | (b) 着手可否の判定 |
| 提言・抜けもれ | 任意 | 暗黙サブタスク（i18n・両OS・回帰・後方互換・制約）の先回り | (b) 段取りの補強 |
| サブタスク | 必須 | `- [ ] ...<br>...` のチェックリスト。誰が→何を→どの品質ゲートの段取りを並べる | **(b) ワークフローの本体（段取り）** |
| 次アクション | 必須 | 今すぐ着手すべき次の一手。担当→段取りの流れで書く | **(b) ワークフローの入口** |
| 更新日 | 必須 | 最終更新日（YYYY-MM-DD） | — |

**(b) 完了までのワークフロー = 担当 ＋ サブタスク ＋ 次アクション ＋ 受け入れ条件（DoD）の合算で表現する。** 新欄は足さない。

品質ゲートの明示ルール（新欄を足さず既存欄に必ず織り込む）:
- サブタスクの最後に検証ステップを必ず入れる。例: `- [ ] tsc --noEmit / eslint / test green<br>- [ ] reviewer 独立検証<br>- [ ] test-functional 実機検証 → REVIEW→DONE`
- コード変更は tsc/eslint/test、reviewer 独立レビュー、test-functional 実機検証を段取りに含める（[[feedback-quality-efficiency-accuracy]] の生成→検証→統合）。
- server コード変更は restart 必須・web は dist 再ビルドをサブタスクに明記。
- 「品質ゲート」を独立欄にしたい場合のみ任意で追加してよいが、原則はサブタスク内に書く（既存カード互換のため）。

## 3. コピペ用 空テンプレ

縦型カード形式。既存台帳と完全互換。`### <ID> — <タイトル>` 見出しの直下にこの表を置く。

```markdown
### MC-XX — タイトル（動詞句で着地が分かる形）

| フィールド | 値 |
|---|---|
| ID | MC-XX |
| タイトル | タイトル（一行で何をやるか。着地が分かる動詞句） |
| 優先度 | P2 |
| ステータス | TODO |
| 担当 | dev-logic（実装）+ designer（UI、必要時） |
| 詳細 | 【(a) 何をやっているか】背景・現状・解決したい課題・着地イメージ。Keita 投入原文があれば「【Apollo投入】...」で引用。 |
| 関連 | 関連ファイル（path:line）・関連タスク（MC-xx）・参照 doc |
| 受け入れ条件（DoD） | (1) 検証可能な完了条件。(2) ... (3) 回帰なし・後方互換・両OS・i18n 等の暗黙条件も明示。 |
| 依存 | 前提タスク・着手条件。無ければ「なし」 |
| 提言・抜けもれ | 暗黙サブタスク（テスト・回帰・後方互換・制約・UI chrome ルール等）を先回りで列挙。 |
| サブタスク | - [ ] 【(b) 段取り1】誰が何をやる<br>- [ ] 段取り2<br>- [ ] 段取り3<br>- [ ] 品質ゲート: tsc/eslint/test green<br>- [ ] reviewer 独立検証<br>- [ ] （server変更時）restart / （web）dist 再ビルド<br>- [ ] test-functional 実機検証 → REVIEW→DONE |
| 次アクション | 担当が着手する最初の一手 → 次の段取り → 検証 の流れ |
| 更新日 | YYYY-MM-DD |
```

サマリ表にも1行追加する（collector はこの表行のステータスを正本に読む）:

```markdown
| MC-XX | タイトル | P2 | 区分/フェーズ | TODO | dev-logic | 依存 |
```

## 4. 記入例（MC-82 自身を題材に）

実在の MC-82 を「良い記入例」として示す。詳細（何をやっているか）と、完了までのワークフロー（担当→段取り(サブタスク)→品質ゲート→DONE）が読める形。

```markdown
### MC-82 — タスクボードの各タスクに「詳細＋成果物完了までのワークフロー」を記載する運用整備

| フィールド | 値 |
|---|---|
| ID | MC-82 |
| タイトル | Apollo タスクボードの各タスクに「何をやっているか詳細＋成果物完了までのワークフロー」を記載する運用整備 |
| 優先度 | P2 |
| ステータス | IN_PROGRESS |
| 担当 | task-manager（記載運用ルール・テンプレ整備）+ designer/dev-logic（TaskDetail での表示） |
| 詳細 | 【Apollo投入】アポロのタスクボードのタスクは何をやっているか詳細を記載すること → 成果物完了までのワークフローも記載すること。各タスクに (a) 今何をやっているか（詳細）、(b) 成果物が完了するまでのワークフロー（誰が→何を→どの品質ゲート→DONE までの段取り）を残す。台帳側はフィールド充実で、Apollo 側は TaskDetail（MC-83）でそれを見やすく表示する両輪。 |
| 関連 | docs/TASK_AUTHORING_TEMPLATE.md（本テンプレ）, docs/TASK_TRACKER.md, MC-83（TaskDetail 表示・DONE）, MC-71（手動編集の md 書き戻し） |
| 受け入れ条件（DoD） | (1) タスク記載のテンプレ/運用ルールが定まる（詳細＋ワークフロー＝担当・段取り・品質ゲート・DoD を必須欄化）。(2) 新規起票時に task-manager がこのテンプレで構造化する運用が回る。(3) Apollo の TaskDetail で「詳細」と「ワークフロー（サブタスク/段取り）」が読める（MC-83 で実装済）。 |
| 依存 | MC-83（表示面、DONE 済）。MC-71（Apollo から詳細編集するなら連動）。 |
| 提言・抜けもれ | 新フィールド乱立を避け既存欄を徹底活用。全タスク遡及は重いので新規から必須・既存は優先度順に追記。UI chrome 制約（中立丁寧体/CSS変数/SVG のみ）は表示側 MC-83 の領分。 |
| サブタスク | - [ ] task-manager が docs/TASK_AUTHORING_TEMPLATE.md を作成（必須欄・空テンプレ・記入例・運用ルール）<br>- [ ] 既存カード互換を確認（MC-82/83/84 の縦型カードに合致）<br>- [ ] 新規起票でこのテンプレを適用する運用に切替<br>- [ ] 既存タスクへ優先度順に詳細＋ワークフローを追記<br>- [ ] 林（オーケストレーター）が確認・台帳ステータス更新・commit |
| 次アクション | task-manager がテンプレを確定 → 新規起票から必須適用 → 既存は優先度高い順に追記 |
| 更新日 | 2026-06-01 |
```

読み取れること:
- (a) 詳細欄に「何をやっているか」（Keita 原文引用＋背景）が入っている。
- (b) ワークフロー = 担当（task-manager 起点）→ サブタスク（テンプレ作成→互換確認→運用切替→既存追記→確認）→ 品質ゲート（互換確認・林の確認）→ DONE（DoD 1〜3 充足）の流れが読める。

実装系タスクの良い例は実在の MC-84 を参照（サブタスクに「セレクタ追加→inbox priority→引き継ぎ→凡例→restart→test-functional 実機検証」と段取り＋品質ゲートが並び、DONE note に reviewer/test の検証根拠が file:line で残っている）。

## 5. 運用ルール

新規起票・既存追記・整合性維持のルール。

**新規起票（task-manager が構造化）**
- 新しい actionable（Keita 依頼 / inbox / フィードバック / 調査で判明した修正 / 施策）が出たら、着手前に必ず task-manager に通してこのテンプレで起票する（[[feedback-route-all-to-task-manager]] / [[feedback-taskboard-based-execution]]）。
- 採番は必ず `bash /home/dev/cron-scripts/next-task-id.sh <PREFIX>`（目視で数えない。[[reference-task-id-numbering]]）。起票は直列化し、起票前に `git pull --rebase`。
- 必須欄（§2 で必須のもの）を空欄にしない。特に 詳細・サブタスク・次アクション・受け入れ条件 は (a)(b) の核なので必ず埋める。

**既存タスクへの追記**
- 全タスク遡及適用は重いので、新規起票から必須適用し、既存は優先度高いもの（P0→P1→…）から順に 詳細＋ワークフロー を追記する。
- 追記時も既存の縦型カード構造を壊さず、欠けている欄を足す形で行う。

**整合性（最重要）**
- サマリ表（`| ID | タイトル | 優先度 | ... | ステータス | 担当 | 依存 |`）の行が正本。collector は表行のステータスを読む。縦型カードのステータスと表行を食い違わせない。
- ステータス変更は必ず docs/TASK_TRACKER.md に反映してから報告。表行とカード両方を同時に更新する。
- 編集は `git pull --rebase` 後に該当行のみ。`git add` は名指し（autonomous-rin / 他アクターとのレース・巻き込み回避。[[feedback-all-agents-taskboard-based]]）。

**役割分担**
- task-manager（棚町）= 台帳の正本管理（登録・構造化・完了検証・抜け漏れ提言）。実装はしない。
- 実装/検証エージェント = 自分が触るタスクのステータスを正しく上げる（着手で IN_PROGRESS、完了で DONE/REVIEW、検証根拠を note に file:line で残す。[[feedback-review-agent-verify-then-done]]）。
- apollo番人 = 抜け漏れ・遅延の監視。
- 林（オーケストレーター）= 台帳ステータス更新・commit はここが行う。
