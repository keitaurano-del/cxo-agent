# ドキュメント・テンプレート（様式）ライブラリ機能 設計提案 (2026-06-09)

Keita「いろんな資料からドキュメンテーションのテンプレートを作成し、用途毎に“この用途にはこのテンプレ”とパターン化できる機能」の設計案。RAG（資料Q&A）とは別機能。実装前の方針合意用。

---

## 1. 狙い（何を・なぜ）

Keita の業務は**標準化された成果物を量産するコンサル/PM支援**（PMBOK系の定義書・設計書・報告書・計画書・WBS・議事録 等。pptx 360 / xlsx 285 / docx 30）。既に `PM人材育成支援/標準様式`・`標準フレームワーク` という**様式資産**を持つ。

そこでこの機能の本質は **「散在する実資料・既存様式を“使える雛形カタログ”に変え、用途（文書種別・工程・目的）で引けて、“この用途にはこれ”と提案し、すぐ着手できる状態にする」** こと。ゼロからAIで雛形を捏造するより、**実在の優れた様式を資産化・パターン化**する方が圧倒的に価値が高い。

提供価値：
- **探さなくていい** — 「確認結果報告を書きたい」→ 最適な様式が出る
- **品質が揃う** — 毎回ベストな型から始められる（属人化解消）
- **着手が速い** — 空の雛形（インスタンス内容を抜いた版）がすぐ手に入る

---

## 2. 中核コンセプト：テンプレート＝「型 ＋ 用途タグ ＋ 記入ガイド ＋ 出典」

1テンプレートは「ファイルの体裁」だけでなく **メタ情報込み**で価値が出る：

```
Template {
  id
  name            例: 確認結果報告書 / アクター定義書 / コミュニケーション管理計画
  docType         文書種別（報告書 / 定義書 / 設計書 / 計画書 / 議事録 / 提案書…）
  useCases[]      用途タグ（いつ・何のために使うか。複数可）
  process         工程（企画 / 要件 / 設計 / 開発 / 移行 / 運用 …）※PMBOK/自社工程に対応
  format          docx / xlsx / pptx / md
  structure       型＝見出し/章/表レイアウト（スケルトン）
  fieldGuidance   各セクションに「何を書くか」の記入ガイド
  sourceDocs[]    元になった実資料（出典・トレーサビリティ）
  blankFile?      インスタンス内容を抜いた“空の様式”ファイル（あれば）
  sampleExcerpt?  記入済み実例の抜粋（参考）
}
```

「用途毎にこれを使うといいですよ」は、この `useCases`/`process`/`docType` を引いた**推薦**で実現する。

---

## 3. 全体パイプライン

```
[A 取り込み/解析] 実資料・既存様式を読む → 構造(見出し/章/表/フィールド)を抽出
        ↓
[B 分類/クラスタリング] 文書種別・工程で分類。似た文書をまとめ「型」を見つける
        ↓
[C テンプレ抽出] 各型の「共通スケルトン＋記入ガイド」を生成。インスタンス内容は除去
        ↓        （※実ファイル体裁を保つ“空様式”生成は上位フェーズ。§5の分岐）
[D 用途タグ付与] docType / useCases / process / format をメタ付与（AI＋人手確定）
        ↓
[E カタログ化] 検索・絞り込み可能なライブラリに登録（用途/工程/種別/フォーマットで引ける）
        ↓
[F 推薦] ユーザの意図（「顧客向け確認結果報告を書きたい」）→ 最適テンプレを提案
        ↓
[G 利用] テンプレ選択 → 空の雛形を取得（or AIが文脈から下書き） → 編集 → 成果物
```

## 4. UI/UX（イメージ）

- **テンプレ・カタログ画面**: カード一覧（名称・文書種別・用途・工程・フォーマット）。フィルタ＝種別/工程/フォーマット/フリーワード。
- **テンプレ詳細**: 型（章立て）プレビュー＋記入ガイド＋出典＋「空の様式をダウンロード/この型で作成」。
- **用途から探す**: 「何を作りたい？」入力 → 推薦テンプレを理由付きで提示（“この用途にはこれ”）。
- 既存 Deliverables（成果物）画面に「テンプレートから作成」を統合してもよい。

## 5. 設計の分岐点（ここがKeita確認の核心）

### ★分岐1：テンプレの“実体”をどこまで作るか（最重要）
- **(a) 構造テンプレ（軽量）**: 型＝見出し/章立て＋記入ガイドを Markdown 等で持つ。AIで抽出しやすく早い。が、pptx/xlsx の細かな体裁は再現しない。
- **(b) 様式そのものをカタログ化（実用的・推奨の起点）**: 既存の `標準様式` 実ファイル（pptx/xlsx）を**そのままテンプレ登録**し、用途タグ＋ガイドを付ける。「空の様式が欲しい」に直結。AIの役割は分類・タグ・ガイド生成・推薦。
- **(c) 様式の自動生成（高度）**: 記入済み実資料から**インスタンス内容を抜いた“空の様式”を自動生成**（体裁保持）。最も価値が高いが、pptx/xlsx の安全なプレースホルダ化は難所。

→ **おすすめ順序**: まず(b)で「実在様式を資産化・用途付け・推薦」を立ち上げ、(a)の構造ガイドを併載。(c)は上位フェーズで対象を絞って。

### ★分岐2：素材コーパス
- どこを母集団にする？ `PM人材育成支援/標準様式`＋`標準フレームワーク`を一次対象にするのが筋が良い。加えて過去成果物全体／notebook資料も取り込むか。

### ★分岐3：用途タクソノミ（分類軸）
- 事前定義（工程×文書種別。PMBOK/自社標準に合わせる）か、AIに資料から**自動で分類軸を立てさせる**か（emergent）。実務に合うのは「自社の工程・様式体系に沿った事前定義＋AI補完」。

### ★分岐4：推薦の賢さ
- タグ一致の素朴な絞り込みから始め、後でAI（意図理解→様式マッピング）に。

### ★分岐5：自動 vs 人手確定
- AIが「これは確認結果報告書／用途はXX」と**提案 → Keita が確定**してライブラリ登録（品質担保）。全自動より、提案→承認が安全。

## 6. 段階ロードマップ

**Phase 1（資産化・カタログ・推薦の最小実用）**
- `標準様式`等の実ファイルを走査 → AIが**文書種別・用途・工程を分類してタグ付け（提案）→ Keita 確定** → カタログ登録（メタ＋出典＋実ファイルへのリンク）
- カタログ画面（一覧/フィルタ）＋「用途から探す（タグ推薦）」
- 各テンプレに記入ガイド（AI生成）

**Phase 2（着手支援）**
- 「この型で作成」＝空の雛形を提供（(b)実ファイルの複製 or (a)構造スケルトン）。AIが文脈（関連資料）から**下書き自動生成**するオプション
- 推薦をAI意図理解に（「〜を作りたい」→最適様式）

**Phase 3（高度な様式生成）**
- 記入済み実資料から“空の様式”を自動生成（(c)・体裁保持・プレースホルダ化）。対象種別を絞って段階導入
- 利用統計でテンプレ改善（よく使う型・記入のつまずき）

---

## 7. 既存資産との関係

- **議事録テンプレ（minutesPresets）の一般化版**: あの id/label/body/用途 を、全文書種別＋実資料由来＋用途推薦に拡張したもの。議事録はこのカタログの1カテゴリに収斂できる。
- **RAG とは別**: RAGは「資料の中身を検索して答える」。本機能は「資料の“型”を抽出して再利用する」。母集団は共有でも目的が違う。
- **Deliverables 基盤を活用**: 既存の成果物保存/プレビュー/ファイル変換（office→PDF）をそのまま使える。

## 8. Keita 確認事項（これだけ決めれば設計を実装計画に落とせます）

1. **分岐1**: まず **(b)「既存の標準様式を“用途タグ付きカタログ”に資産化」**から始めてよいか（推奨）。(c)空様式の自動生成はPhase3で。
2. **分岐2**: 一次コーパスは `PM人材育成支援/標準様式`＋`標準フレームワーク` でよいか。他に含めたい資料源は？
3. **分岐3**: 分類軸は「自社の工程×文書種別」を事前定義したい（その体系を教えてください）か、AIに資料から立てさせるか。
4. **用途の主目的**: 自分用の効率化か、チーム/顧客にも配る資産か（共有範囲で作り込みが変わる）。
5. **着手順**: Phase1（カタログ＋推薦）から進めてよいか。

特にこだわり無ければ私の判断（(b)から・標準様式を一次対象・工程×種別の事前定義＋AI補完・提案→Keita確定）で Phase1 を設計→実装します。
