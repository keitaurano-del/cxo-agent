現行実装の徹底棚卸し（Son調査）に基づく。実装前の方針合意用ドキュメント。

---

## 1. 「本来のRAG」とは — 目的の再定義

RAG（Retrieval-Augmented Generation）の本質は **「LLMに、根拠資料を“正しく見つけて”“正しく渡し”、根拠に基づいた答えを“出典付きで”返させる」** 仕組み。良いRAGの条件は4つ：

1. **検索が当たる（Retrieval品質）** — 質問に本当に関係するチャンクが上位に来る。無関係な断片を渡さない。
2. **根拠に忠実（Grounding）** — 渡した資料の範囲で答える。資料に無いことは「資料に無い」と言う（幻覚抑制）。
3. **出典が辿れる（Transparency）** — どのファイルのどの箇所が根拠かをユーザが確認できる。
4. **回り続ける／改善できる（Operability）** — 規模が増えても速く、品質を測って継続改善できる。

この cxo-agent の RAG の用途は **「アップロードした業務資料に対する Q&A ＋ 生成物（要約/FAQ/時系列/雛形）作成」**。社内ナレッジ・プロジェクト資料が対象（現状222資料/3116チャンク規模のNBあり）。よって上記4条件、特に **①検索品質・③出典・④規模耐性** が効く。

## 2. 現状アーキテクチャ（要約）

取り込み(多形式→テキスト抽出) → チャンク分割(800字/100重複・段落優先) → 埋め込み(Gemini gemini-embedding-001・**768次元**) → 保存(NBごと `index/chunks.json` にベクトル同梱・**122MBの巨大JSON**) → 検索(クエリ埋め込み→**全チャンクとコサイン総当たり**→top5) → 生成(`claude -p` sonnet→opusフォールバック・抜粋をプロンプトに埋め込み・`{{cite:file:chunkIdx}}` 出典) → 3ペインUI。

## 3. 構造的な弱点（影響度順）

| # | 弱点 | 影響 |
|---|---|---|
| A | **検索が素のコサインtop5のみ**（スコア閾値なし・リランクなし） | 関連薄いチャンクも混入→プロンプト希釈→回答品質低下。逆に本当に効く断片が5枠から漏れる |
| B | **ベクトル検索のみ（ハイブリッド無し）** | 固有名詞・略語・ID（"MC-202","PMBOK","型番"）の完全一致に弱く取りこぼす |
| C | **巨大JSON総当たり**（122MB chunks.json を毎クエリ全読み・全件内積） | 規模拡大で線形に遅化・メモリ圧迫。NBが増える/大きいほど破綻に近づく |
| D | **チャンク戦略が粗い**（800字固定・メタ情報＝ページ/見出し/章を持たない） | 文脈の途中で切れる・表が壊れる。出典がチャンク番号止まりで「元資料のどこ」が不明 |
| E | **出典がユーザに見えにくい**（回答内タグのみ・使用ソース一覧やページ提示なし） | 「どの資料が根拠か」を後から辿れず信頼性が落ちる |
| F | **生成が `claude -p` CLI依存**（共有アカウント・上限・10分タイムアウト・同時2） | アカウント上限/障害に弱い（直近の Sonnet 上限事故＝MC-202 の主因）。スループット頭打ち |
| G | **品質を測る仕組みが無い** | チャンクサイズ/top-k/モデルを“勘”で調整するしかなく、改善が積み上がらない |

## 4. 再設計（ターゲット像）

原則：**「検索を賢く・根拠を厚く・規模に強く・品質を測れる」**。各コンポーネントの目標形：

### 4.1 検索（最重要）
- **二段検索＋リランク**: ベクトルで広めに候補取得(例 top 20-30) → **リランカで本当に効く top N(5前後)に絞る**。リランクは (a) Cohere/Voyage等のrerank API、または (b) LLM/embeddingベースの軽量再採点、のいずれか。
- **スコア閾値**: 類似度が低いチャンクは捨てる（無関係注入を防ぐ）。閾値割れ＝「資料に該当なし」と素直に返す。
- **ハイブリッド検索**: ベクトル＋**BM25/キーワード**を併用しスコア統合（RRF等）。固有名詞・ID の取りこぼしを解消。

### 4.2 チャンク & メタデータ
- **構造認識チャンク**: 見出し/段落/表を考慮し、章・見出し・**ページ番号・ソースファイル**をチャンクのメタに保持。表は崩さない。
- これにより **出典をページ/章粒度**で出せる（弱点D/E解消の土台）。

### 4.3 ストレージ（規模耐性）
- 巨大JSON総当たりを廃し、**埋め込み型ベクトルストア**へ（候補：SQLite + sqlite-vec / hnswlib / lancedb 等の軽量・サーバ常駐不要なもの）。近似最近傍(ANN)で高速化＋メモリ削減。NB横断検索も視野。
- メタ（source/page/heading/score）を併存。

### 4.4 生成 & 出典提示
- プロンプトは現行の「抜粋＋cite」を踏襲しつつ、**回答末尾に“使用した資料(ファイル＋ページ)”の一覧**を必ず提示。UI で出典をクリック→該当箇所プレビュー。
- **根拠不足時の明示**（資料に無ければ「資料に無い」と返す）をプロンプト＋閾値で担保。
- 生成エンジン：当面 `claude -p`（urano2/Opus）でよいが、**上限/障害に強い経路**（API直/別アカ/キュー）を中期で検討。

### 4.5 評価・改善ループ（地味だが効く）
- **ゴールデンQ&Aセット**（代表質問＋期待出典）を用意し、(a)検索ヒット率(該当チャンクがtop-Nに入るか)、(b)回答の根拠忠実度 を定点観測。
- チャンクサイズ/top-k/リランク有無/モデルを**数値で比較**して調整できる土台を作る。UIに簡単な👍👎フィードバックも。

## 5. 段階的ロードマップ（小さく効く順）

**Phase 1（即効・低リスク・既存構造のまま品質UP）**
- スコア閾値導入（無関係チャンク除去・該当なし時は正直に返す）
- 二段検索＋リランク（top20→rerank→top5。まずはembedding/LLM軽量rerankでAPI追加なしでも可）
- 回答に**使用ソース一覧（ファイル名）**を明示＋UI表示
- ハイブリッド（BM25）の素朴版を追加

**Phase 2（検索の芯を強くする）**
- 構造認識チャンク＋メタ（ページ/見出し/ソース）→ **出典をページ粒度**に
- リランカを本格化（rerank API 比較検討）
- ゴールデンQ&A＋検索ヒット率の評価ハーネス

**Phase 3（規模と土台）**
- ベクトルストア移行（sqlite-vec 等）で巨大JSON総当たりを廃止・高速化・NB横断
- 生成エンジンの上限/障害耐性（API/キュー/別経路）

## 6. 着手前に決めたい点（Keita 確認事項）

1. **主用途の優先**: この RAG は主に「①社内資料のQ&A」か「②生成物(要約/FAQ/雛形)作成」か。どちらを優先して品質を上げるかで検索/プロンプトの寄せ方が変わる。
2. **外部API追加の可否**: リランク/埋め込みで Cohere/Voyage 等の有料APIを足してよいか（精度↑だがコスト）。不可なら Gemini＋自前リランクで組む。
3. **ベクトルストア導入の可否**: Phase3 で SQLite拡張等の依存追加を許容するか（規模耐性に直結）。
4. **スコープ/順序**: Phase1から順に回すか、特に困っている点（例：回答が浅い／出典が辿れない／特定資料が当たらない）を先に潰すか。具体的に「ここが不満」があれば最優先にする。
